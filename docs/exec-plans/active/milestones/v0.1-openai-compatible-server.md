# v0.1 OpenAI-Compatible Server

## Goal

Expose the v0.0 generator through OpenAI-compatible HTTP endpoints.

## Why

Production serving starts with a stable API that existing tools can call.

## Scope

- FastAPI server
- `/v1/models`
- `/v1/responses`
- `/v1/completions`
- `/v1/chat/completions`
- SSE streaming

## Out Of Scope

- continuous batching
- auth
- multi-model routing
- Responses tools, multimodal inputs, background runs, and cancellation
- Assistants API

## Acceptance Criteria

- `curl /v1/chat/completions` returns a valid response.
- `curl /v1/responses` returns a valid response.
- Streaming emits incremental tokens.
- Basic request/response protocol types are covered by tests.

## Notes

OpenAI-compatible endpoints make the teaching engine usable with ordinary
client tooling before the internal scheduler and cache become sophisticated.
The implementation intentionally adapts the v0.0 single-request generator rather
than introducing routing, authentication, tools, or batching early.

The API priority is:

- `/v1/responses`: preferred new-project surface. nanoLLMServe supports a
  text-only subset with `input`, `instructions`, `max_output_tokens`,
  `output_text`, usage, streaming deltas, and lightweight in-memory retrieval.
  Full lifecycle, cancellation, tool-call, and event parity is deferred to
  [`v0.10 Responses API parity`](./v0.10-responses-api-parity.md).
- `/v1/chat/completions`: supported compatibility surface for chat-shaped
  clients and provider interoperability.
- `/v1/completions`: supported legacy prompt-to-completion surface.
- Assistants API: not supported; vLLM/SGLang do not expose Assistants routes in
  the checked reference trees, and the modern direction is Responses.

Streaming is implemented as server-sent events over the same naive decode loop.
Each generated token yields one chunk, and the stream ends with `data: [DONE]`.
The `/v1/completions` stream uses `object=text_completion`, matching vLLM and
SGLang. The `/v1/chat/completions` stream uses `object=chat.completion.chunk`.

The server validates the request `model` against the single served model and
returns `model_not_found` for unknown names, mirroring the reference servers'
single-model contract. vLLM and SGLang both implement `/v1/responses`,
`/v1/responses/{response_id}`, and `/v1/responses/{response_id}/cancel`; this
milestone implements create and retrieve, while leaving background execution and
cancel for a future agent/tool milestone.

## Validation Log

- `python3 -m pytest -q` on the local no-FastAPI/no-torch environment:
  `8 passed, 4 skipped`.
- `PYTHONPATH=src /data/anaconda3/bin/python -m pytest -q` on the A100
  validation host: `37 passed`.
- Real-model A100 smoke with cached Qwen3-0.6B:
  - `curl /v1/models` returned `Qwen3-0.6B`.
  - `curl /v1/responses` returned `object=response` with `output_text` and
    usage.
  - `curl /v1/responses/{response_id}` retrieved the stored response.
  - `curl /v1/chat/completions` returned `object=chat.completion` with usage.
  - `curl -N /v1/responses` returned Responses SSE events including
    `response.output_text.delta` and `response.completed`.
  - `curl -N /v1/completions` returned incremental `data:` chunks followed by
    `data: [DONE]`, with `object=text_completion`.
  - `curl /v1/completions` with an unknown `model` returned `model_not_found`.
