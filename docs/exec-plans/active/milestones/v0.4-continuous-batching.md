# v0.4 Continuous Batching

## Goal

Let requests enter and leave the active batch while the engine is running.

## Why

Continuous batching is a core vLLM/SGLang serving idea. It avoids waiting for the slowest request in a static batch.

## Scope

- `RequestState`
- scheduler
- waiting queue
- running queue
- per-step batch rebuild
- request completion and removal

## Out Of Scope

- paged KV cache
- chunked prefill
- fairness policies beyond a simple first version

## Acceptance Criteria

- New requests can be admitted while other requests are decoding.
- Completed requests leave the running set without stopping the engine.
- Metrics show active batch size over time.

Current implementation statement:

- `generate_continuous_batch` implements scheduler-level continuous batching by
  admitting requests by `arrival_step`, rebuilding a padded full-token active
  batch each scheduler step, and removing finished rows after the step.
- This milestone deliberately rebuilds full-token batches instead of carrying
  per-row paged KV cache. Paged/block KV cache ownership starts in `v0.5`.
