# v0.5 Block KV Cache Manager

## Goal

Manage KV cache in token blocks instead of one large contiguous tensor per request.

## Why

Block management introduces the main motivation behind PagedAttention: reduce KV cache fragmentation and improve memory use.

## Scope

- configurable block size, initially 16 or 32 tokens
- free block pool
- request-to-block table
- simple block allocation and release
- simplified paged KV model

## Out Of Scope

- full vLLM PagedAttention kernel
- GPU custom block kernels
- prefix cache

## Acceptance Criteria

- Requests allocate and release KV blocks.
- KV block usage is observable.
- Fragmentation behavior is demonstrated with a small benchmark.

Current implementation statement:

- `KVBlockManager` tracks fixed-size token blocks, a FIFO free block pool,
  per-request block tables, allocation/release, and usage metrics.
- `generate_one`, `generate_batch`, and `generate_continuous_batch` can receive
  a `KVBlockManager` to allocate prompt blocks, append generated-token capacity,
  and release blocks at completion.
- This is allocator metadata for teaching PagedAttention motivation; actual KV
  tensors still come from Hugging Face `past_key_values`, and GPU block kernels
  remain out of scope.
