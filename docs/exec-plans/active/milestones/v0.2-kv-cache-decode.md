# v0.2 KV Cache Decode

## Goal

Split prefill and decode so generation reuses KV cache.

## Why

KV cache is the line between naive repeated forward passes and real LLM inference.

## Scope

- `past_key_values`
- prefill/decode separation
- decode with only the last token
- TTFT and TPOT measurements

## Out Of Scope

- paged KV cache
- prefix cache
- custom attention kernels

## Acceptance Criteria

- Generation reuses KV cache after prefill.
- Benchmark reports TTFT and TPOT.
- A before/after comparison against v0.0 exists.

## Design Notes

`v0.2` intentionally uses Hugging Face `past_key_values` directly instead of
introducing a paged KV cache or block allocator. The shape mirrors the smallest
useful idea from the reference systems:

- vLLM tracks per-request computed context, such as `num_computed_tokens`, so
  scheduling can avoid recomputing tokens that already have KV state.
- SGLang separates `extend` / prefill from `decode`, and benchmarks those stages
  separately.
- `nanoLLMServe` keeps the teaching version single-request: prefill receives the
  full prompt, samples the first token, and every later decode step receives only
  the previous sampled token plus cached `past_key_values`.

This milestone makes the performance boundary visible while leaving paged KV
cache, block allocation, prefix reuse, and batching to later milestones.
