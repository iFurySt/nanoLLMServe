# v0.10 Responses API Parity

## Goal

Upgrade the teaching-scale Responses API from a text-only create/retrieve subset
to a stateful agent/tool runtime surface that is deliberately comparable to
vLLM and SGLang.

## Why

`/v1/responses` is the preferred new-project API surface. The v0.1
implementation is intentionally small so the HTTP milestone stays readable, but
leaving it as text-only would make later agent, tool, structured-output, and
multimodal examples depend on the wrong abstraction.

## Scope

- response lifecycle store with status transitions
- `POST /v1/responses/{response_id}/cancel`
- `previous_response_id` continuation semantics
- streaming event sequence closer to OpenAI/vLLM/SGLang
- tool-call response item shape
- structured-output integration where it overlaps with Responses text config
- request and response validation for unsupported fields
- reference comparison against `./tmp/vllm` and `./tmp/sglang`

## Out Of Scope

- full production persistence
- distributed response store
- complete OpenAI parity for every tool type
- browser/computer/code interpreter execution
- multimodal inputs unless the earlier roadmap has introduced multimodal model
  support

## Acceptance Criteria

- `POST /v1/responses` supports the response object fields needed by
  text, tool-call, and structured-output examples.
- `GET /v1/responses/{response_id}` returns stored state for completed and
  in-progress responses.
- `POST /v1/responses/{response_id}/cancel` transitions cancellable responses
  to `cancelled`.
- `previous_response_id` can continue from a stored response without losing
  prior conversation context.
- Streaming emits ordered Responses events with stable IDs and sequence numbers.
- Tests cover create, retrieve, cancel, continuation, streaming, unsupported
  fields, and at least one tool-call shaped output path.
- Docs explain which parts intentionally match vLLM/SGLang and which parts
  remain teaching-scale.
